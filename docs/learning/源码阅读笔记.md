# 源码阅读笔记：从 H5 send 到 ArkTS 回调 H5

这篇文档解决的问题：用一次 `window.myascf.send(...)` 调用，串起 MiniAppRuntime-Harmony 的完整 JSBridge 调用链路，帮助理解 H5、ArkWeb、JavaScriptProxy、BridgeController、Dispatcher、Registry、Biz、Imp、CallbackExecutor 和 callback map 之间的关系。

---

## 1. entry 通过 ArkWeb 加载 H5，并注入 NativeProxy

在 `entry` 示例应用中，页面首先创建 ArkWeb 的 `WebviewController`：

```ts
private controller: webview.WebviewController = new webview.WebviewController()
```

`WebviewController` 用于控制 ArkWeb 页面，后续 ArkTS 回调 H5 时，会通过它执行：

```ts
controller.runJavaScript(...)
```

同时，`entry` 创建 `MyASCFRuntime`：

```ts
private runtime: MyASCFRuntime = new MyASCFRuntime(
  this.controller,
  getContext(this)
)
```

这里两个参数职责不同：

```text
controller：负责 Web 页面控制和 runJavaScript 回调 H5。
context：表示当前宿主应用的运行环境，Storage 等能力需要通过它访问应用本地存储。
```

然后在 `Web()` 组件中注册 JavaScriptProxy：

```ts
.javaScriptProxy({
  object: this.runtime.getNativeProxy(),
  name: this.runtime.getProxyName(),
  methodList: this.runtime.getMethodList(),
  controller: this.controller
})
```

这样 H5 页面中就可以访问：

```js
window.MyASCFNative.postMessage(...)
```

这里的 `MyASCFNative` 不是 H5 自己创建的，而是 ArkTS 通过 `.javaScriptProxy(...)` 暴露给 H5 的对象。

这一层可以理解为：

```text
entry 负责加载 H5 页面，并把 HAR runtime 中的 NativeProxy 挂到 H5 的 window 上。
```

---

## 2. H5 调用 window.myascf.send

H5 页面点击按钮后，会调用：

```js
window.myascf.send("ui.showToast", {
  message: "hello from h5"
})
```

`send` 方法负责创建一次 JSBridge 请求。

它会生成 `requestId`，并拼出请求对象：

```js
{
  requestId,
  action,
  params
}
```

其中：

```text
requestId：标识一次请求。
action：表示要调用的 Native API，例如 ui.showToast。
params：表示本次调用携带的参数。
```

`requestId` 的作用是保证多个并发请求返回时，H5 能知道哪个响应对应哪个请求。

例如同时发出多个请求：

```js
window.myascf.send("ui.showToast", { message: "A" })
window.myascf.send("system.clipboard.readText", {})
window.myascf.send("system.storage.getItem", { key: "username" })
```

它们返回的顺序不一定和发送顺序一致，所以必须通过 `requestId` 关联请求和响应。

---

## 3. H5 保存 callback

在真正发送消息前，H5 会把当前 Promise 的 `resolve`、`reject`、`timer` 等信息保存到 callback map 中。

可以理解为：

```text
requestId -> callback
```

也就是：

```text
发送请求前：保存 callback。
收到响应后：根据 requestId 查 callback。
处理完成后：删除 callback。
```

这一步解决的是异步回调对应关系问题。

如果没有 callback map，H5 收到 ArkTS 回调时，就不知道应该 resolve 哪一个 Promise。

---

## 4. H5 调用 window.MyASCFNative.postMessage

当请求对象和 callback 都准备好后，H5 调用：

```js
window.MyASCFNative.postMessage(JSON.stringify(request))
```

这一步会把 JSON 字符串从 H5 侧发送到 ArkTS 侧。

`window.MyASCFNative` 是 ArkTS 通过 `.javaScriptProxy(...)` 暴露出来的对象。

所以这一步可以理解为：

```text
H5 把 JSBridge 请求序列化成字符串，然后通过 ArkWeb JavaScriptProxy 发送给 ArkTS。
```

---

## 5. JavaScriptProxy 接收消息

ArkTS 侧的 `JavaScriptProxy` 接收到 H5 传来的字符串。

它本身不处理具体业务，只负责把消息转交给 `BridgeController`：

```ts
this.bridgeController.handleMessage(message)
```

这里体现了一个设计原则：

```text
JavaScriptProxy 是通信入口，不是业务处理层。
```

它不应该做：

```text
不判断 action。
不校验业务参数。
不调用 Toast / Clipboard / Storage。
不拼接 runJavaScript 回调。
```

它只负责把 H5 消息带入 ArkTS runtime。

---

## 6. BridgeController 解析请求

`BridgeController` 收到原始字符串后，会尝试解析 JSON。

如果 JSON 解析失败，返回 `PARSE_ERROR`。

如果解析成功，就得到 `BridgeRequest`：

```ts
{
  requestId,
  action,
  params
}
```

然后 `BridgeController` 把请求交给 `BridgeDispatcher`：

```ts
const response = await this.dispatcher.dispatch(request)
```

Controller 不直接判断 action，也不直接调用 Toast、Clipboard、Storage 等能力。

它的职责应该保持简单：

```text
接收 raw message。
解析 BridgeRequest。
调用 BridgeDispatcher。
拿到 BridgeResponse。
交给 BridgeCallbackExecutor 回调 H5。
```

如果把 action 判断、参数校验、系统 API 调用和回调拼接都写在 Controller 里，Controller 会越来越重，后续每新增一个 API 都要改 Controller。

---

## 7. BridgeDispatcher 分发 action

`BridgeDispatcher` 根据请求中的 `action` 找对应 handler：

```ts
const handler = this.registry.get(request.action)
```

如果找不到 handler，返回：

```text
UNKNOWN_ACTION
```

如果找到了，就执行 handler。

如果 handler 执行过程中抛出异常，统一返回：

```text
INTERNAL_ERROR
```

Dispatcher 的职责是：

```text
根据 action 找 handler。
执行 handler。
统一处理 UNKNOWN_ACTION。
统一兜底 INTERNAL_ERROR。
```

它不负责参数校验，也不直接调用系统能力。

---

## 8. HandlerRegistry 查找 handler

`HandlerRegistry` 负责保存 action 和 handler 的映射关系。

这些 handler 不是请求来了才注册的，而是在 runtime 初始化时由 `RuntimeBootstrap` 注册进去的。

例如：

```text
ui.showToast -> toastBiz.handle
system.clipboard.writeText -> clipboardBiz.writeText
system.clipboard.readText -> clipboardBiz.readText
system.storage.setItem -> storageBiz.setItem
system.storage.getItem -> storageBiz.getItem
system.storage.removeItem -> storageBiz.removeItem
system.storage.clear -> storageBiz.clear
```

所以请求到达时，Registry 只负责查找，不负责业务逻辑。

这里要特别注意：

```text
RuntimeBootstrap 是初始化阶段使用的。
HandlerRegistry 是请求阶段查询 handler 用的。
```

也就是说：

```text
应用启动 / runtime 初始化时：
RuntimeBootstrap 把 action 和 handler 注册进 HandlerRegistry。

H5 发起请求时：
BridgeDispatcher 通过 HandlerRegistry 根据 action 找到已注册的 handler。
```

---

## 9. Biz 校验参数

找到 handler 后，会进入对应的 Biz 层。

例如 `ToastBiz` 会校验：

```text
params.message 是否存在。
params.message 是否为字符串。
params.message 是否为空。
```

`ClipboardBiz` 会校验：

```text
params.text 是否存在。
params.text 是否为字符串。
params.text 是否为空。
```

`StorageBiz` 会校验：

```text
params.key 是否存在。
params.key 是否为字符串。
params.key 是否为空。
params.value 是否符合当前 API 要求。
```

如果参数不合法，Biz 返回：

```text
PARAM_ERROR
```

Biz 层负责：

```text
参数校验。
业务语义判断。
构造标准 BridgeResponse。
```

Biz 层不负责：

```text
不操作 H5。
不拼接 JS。
不直接管理 callback map。
不直接处理 WebviewController。
```

---

## 10. Imp 调用系统能力

参数校验通过后，Biz 会调用 Imp 层。

例如：

```text
ToastImp -> promptAction.showToast
ClipboardImp -> 剪贴板能力
StorageImp -> Preferences 本地存储能力
```

Imp 层只负责调用 HarmonyOS 公开系统能力。

它不关心：

```text
requestId
action
BridgeResponse
callback map
runJavaScript
DebugPanel
```

这样可以让平台能力调用和 JSBridge 协议解耦。

例如 Storage API 需要使用本地 Preferences 能力，而 Preferences 需要应用上下文 `Context`。所以 `context` 从 entry 传入 `MyASCFRuntime`，再由 runtime 内部传给 StorageImp。

这就是：

```text
宿主应用提供环境。
HAR 框架使用环境。
```

HAR 包本身是库，不是页面，也不是 Ability，所以它不能自己凭空创建应用上下文。

---

## 11. 构造 BridgeResponse

Biz / Imp 执行完成后，会构造统一的 `BridgeResponse`：

```ts
{
  requestId,
  code,
  message,
  data
}
```

成功时 `code` 为 `SUCCESS`。

失败时根据情况返回：

```text
PARAM_ERROR
UNKNOWN_ACTION
INTERNAL_ERROR
PARSE_ERROR
TIMEOUT
CALLBACK_LOST
```

其中：

```text
PARAM_ERROR：参数错误，一般由 Biz 返回。
UNKNOWN_ACTION：action 未注册，一般由 Dispatcher 返回。
INTERNAL_ERROR：内部异常，一般由 Dispatcher 兜底。
PARSE_ERROR：JSON 解析失败，一般由 BridgeController 返回。
TIMEOUT：H5 等待超时，由 H5 侧判断。
CALLBACK_LOST：响应回来时 H5 callback 已不存在，由 H5 侧记录。
```

`BridgeResponse` 的意义是统一 H5 和 ArkTS 之间的响应协议。

无论是 Toast、Clipboard 还是 Storage，最终都应该返回同一种结构，方便 H5 统一处理。

---

## 12. BridgeCallbackExecutor 执行 runJavaScript

`BridgeController` 拿到 `BridgeResponse` 后，不直接拼接 JS，而是交给 `BridgeCallbackExecutor`。

`BridgeCallbackExecutor` 会把 response 序列化，并拼出 H5 回调脚本：

```ts
const script: string = `if (${H5_RESPONSE_CALLBACK}) { ${H5_RESPONSE_CALLBACK}(${responseArg}); }`
```

然后执行：

```ts
await this.controller.runJavaScript(script)
```

这一步把 ArkTS 的处理结果重新送回 H5。

为什么要单独抽出 `BridgeCallbackExecutor`？

因为 ArkTS 回调 H5 时涉及：

```text
BridgeResponse 序列化。
JSON 字符串安全处理。
拼接 H5 回调函数。
执行 runJavaScript。
记录回调日志。
处理回调失败风险。
```

如果这些逻辑散落在 Controller 或 Biz 里，后续很难维护。

所以：

```text
BridgeCallbackExecutor 专门负责 ArkTS -> H5 的回调出口。
```

---

## 13. H5 根据 requestId 查找 callback

H5 收到 ArkTS 回调后，会执行：

```js
window.__myascf_on_native_response__(response)
```

H5 根据 `response.requestId` 去 callback map 中查找之前保存的 callback。

如果找到了：

```text
code === SUCCESS -> resolve(response)
code !== SUCCESS -> reject(response)
```

处理完成后删除 callback。

如果找不到 callback，说明可能已经超时并被删除，这时记录：

```text
CALLBACK_LOST
```

这里要注意：

```text
callback 不是响应回来后才存入 map。
callback 是发送请求前就已经存入 map。
响应回来后只是根据 requestId 查找 callback。
```

所以完整流程是：

```text
发送前：存 callback。
响应后：查 callback。
处理完：删 callback。
```

---

## 14. TIMEOUT 和 CALLBACK_LOST 的关系

H5 侧调用时可以设置 timeout，例如：

```js
window.myascf.send("ui.showToast", {
  message: "hello"
}, {
  timeout: 5000
})
```

这里的 `5000` 表示 5000ms，也就是 5 秒。

TIMEOUT 是 H5 判断的：

```text
H5 发请求。
H5 保存 callback。
H5 启动 timer。
如果 5 秒内收到响应，清理 timer，resolve / reject。
如果 5 秒内没收到响应，删除 callback，reject TIMEOUT。
```

CALLBACK_LOST 是另一种情况：

```text
第 0 秒：H5 发请求。
第 5 秒：H5 超时，删除 callback，Promise reject TIMEOUT。
第 6 秒：ArkTS 才回调 H5。
第 6 秒：H5 根据 requestId 查 callback。
第 6 秒：找不到 callback。
```

这时就是：

```text
CALLBACK_LOST
```

所以：

```text
TIMEOUT 表示 H5 等太久，主动放弃等待。
CALLBACK_LOST 表示 ArkTS 响应回来时，H5 已经找不到对应 callback。
```

## 15. MyASCFRuntime 的作用

`MyASCFRuntime` 是 HAR 包对外暴露的门面类。

外部 Demo 不需要直接创建：

```text
BridgeController
BridgeDispatcher
HandlerRegistry
RuntimeBootstrap
BridgeCallbackExecutor
JavaScriptProxy
ToastBiz
ClipboardBiz
StorageBiz
ToastImp
ClipboardImp
StorageImp
```

外部只需要：

```ts
private runtime: MyASCFRuntime = new MyASCFRuntime(
  this.controller,
  getContext(this)
)
```

然后把 runtime 提供的 NativeProxy 注册给 ArkWeb：

```ts
.javaScriptProxy({
  object: this.runtime.getNativeProxy(),
  name: this.runtime.getProxyName(),
  methodList: this.runtime.getMethodList(),
  controller: this.controller
})
```

这就是门面类的价值：

```text
对外隐藏内部复杂度。
对内完成运行时对象组装。
让新建 Demo 更容易接入 HAR 包。
```

这一步让项目从普通 Demo 更接近可复用框架。

## 16. 整体链路总结

一次完整调用链路是：

```text
H5 button
-> window.myascf.send
-> 生成 requestId
-> 保存 callback
-> window.MyASCFNative.postMessage
-> JavaScriptProxy
-> BridgeController
-> BridgeDispatcher
-> HandlerRegistry
-> Biz
-> Imp
-> BridgeResponse
-> BridgeCallbackExecutor
-> runJavaScript
-> H5 callback map
-> Promise resolve / reject
```

这条链路的核心思想是：

```text
通信入口、action 分发、参数校验、系统能力调用、H5 回调分别由不同模块负责。
```

这样新增 API 时，不需要大改主链路，只需要：

```text
新增 ActionName。
新增 Biz。
新增 Imp。
在 RuntimeBootstrap 中注册。
更新 H5 Demo。
更新 API 文档。
```

---

## 17. 每一层应该记住什么

| 层级 | 主要职责 | 不应该做什么 |
| --- | --- | --- |
| H5 `myascf.send` | 生成 requestId、保存 callback、发送请求 | 不关心 ArkTS 内部怎么分发 |
| JavaScriptProxy | 接收 H5 消息并转发给 Controller | 不处理业务 |
| BridgeController | 解析请求、调用 Dispatcher、回调 H5 | 不写 action if else |
| BridgeDispatcher | 根据 action 找 handler，统一错误兜底 | 不做具体参数校验 |
| HandlerRegistry | 保存 action 与 handler 的映射 | 不调用系统 API |
| RuntimeBootstrap | 初始化时注册内置 API | 不处理每次请求细节 |
| Biz | 参数校验、业务响应结构 | 不拼接 runJavaScript |
| Imp | 调用 HarmonyOS 公开 Kit | 不关心 JSBridge 协议 |
| BridgeCallbackExecutor | 统一 runJavaScript 回调 H5 | 不处理业务参数 |
| MyASCFRuntime | 对外门面，组装内部 runtime | 不暴露过多内部细节 |

## 19. 当前理解总结

这个项目的主线并不复杂：

```text
H5 发请求。
ArkTS 收请求。
Dispatcher 找 API。
Biz 校验参数。
Imp 调系统能力。
ArkTS 回调 H5。
H5 根据 requestId resolve / reject。
```

真正有价值的地方不是“代码能跑”，而是：

```text
每一层职责清楚。
新增 API 不破坏主链路。
错误处理统一。
回调出口统一。
HAR 包可以复用。
外部 Demo 接入简单。
```

这就是 MiniAppRuntime-Harmony 作为一个 HarmonyOS Web 容器与 JSBridge 框架 Demo 的核心意义。
